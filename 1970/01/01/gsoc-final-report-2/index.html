# Background

My first contribution to Open Source software was pretty simple - it was a
documentation fix for a tool I was using. I had some issues building it and
figured that some of the steps were wrong, so I made a PR to fix it. It hit me a
while later that I was probably helping a lot of people out by not letting them
run into the same issue as me, and that felt nice.

I learned about RTEMS when looking for organisations that participated in GSoC
'24. I found the project very interesting and thought it would be cool to
contribute something to it. I received my GSoC acceptance mail on May 2nd, and I
knew from that moment on that this was going to be a unique Summer.

# About the Project

RTEMS (Real-Time Executive for Multiprocessor Systems) is an open-source real-time operating system (RTOS) designed for embedded systems requiring high reliability and deterministic behavior. The project involved modifying GDB to enhance its debugging capabilities within RTEMS.

# Problem Statement

GDB provides support built in to automatically load Python. This is for dynamic
executables only. The way this works is: Whenever a specific library is loaded
by the object file currently loaded in GDB, the linker will trigger GDB to
correspondingly load the pretty-printing scripts for that library.

However, RTEMS is essentially fully statically linked, and the same cannot be said for statically linked executables. There is no
linker in play to let GDB know when a library is loaded, and GDB does not
automatically have Python loaded hence. This causes a problem because now there
is an abundance of features being missed out on due to the inability to load
Python automatically by GDB.

# Project Goals

  • Integrate a Python initializer into GDB for RTEMS to facilitate enhanced debugging.
    modifications to its build
  • Enable GDB to automatically load pretty-printing scripts for RTEMS structures and libstdcxx classes.
    structures
  • Seamlessly integrate these changes into the RTEMS toolchain, reducing the need for user intervention.
    involvement from the user

# Approach

I implemented the addition of a custom section: .debug_gdb_scripts for RTEMS executables,
allowing GDB to execute Python scripts upon program load. This approach was
tailored specifically for RTEMS to overcome limitations with static linking.

# Work Completed

# Python Initializer for GDB

  • Implemented a custom section .debug_gdb_scripts in RTEMS executables for GDB to load Python scripts automatically.
  • Developed and tested integration with the RTEMS build system to ensure Python scripts are executed upon loading an application.

# Pretty-Printers Integration

  • Added support for pretty-printers for RTEMS-specific data structures.
  • Created a script pprinter.py to enhance GDB's ability to display RTEMS structures in a user-friendly format.

# Integration with RTEMS Toolchain

  • Modified the RTEMS Source Builder (RSB) to include the necessary changes for building stdcxx.py with the latest GCC.
  • Updated documentation to reflect new features and usage instructions.

# Current State

  • All major features have been implemented and tested.
  • Python initialization and pretty-printer support have been integrated into the RTEMS build system.
  • Pretty-printers for some basic RTEMS structures have been written and tested.

# Challenges and Learnings

# Challenges

# Integrating Python and GDB with Static Linking in RTEMS

  • RTEMS uses static linking, which complicates GDB's ability to automatically
    load Python scripts, a feature that works for dynamically linked executables.
    This required developing a custom solution to ensure Python scripts could be
    executed within the RTEMS environment

# Developing a Custom Solution for Script Loading

  • The conventional methods for loading Python scripts in GDB did not apply to
    RTEMS. I had to devise a new method to introduce a .debug_gdb_scripts section
    in RTEMS executables
